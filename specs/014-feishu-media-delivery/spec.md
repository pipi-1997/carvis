# 功能规格说明：飞书会话内资源发送

**功能分支**: `014-feishu-media-delivery`
**创建日期**: 2026-03-17
**最近修订**: 2026-03-18
**状态**: 修订中
**输入**: 用户描述："为 carvis 增加飞书会话内的资源发送能力：agent 可以把图片、文件等资源直接发回当前 session；飞书适配器要支持真实资源发送；agent 主路径不能暴露 PATH、cwd、worktree、runId 等底层接线细节。"

## 系统影响 *(必填)*

- **受影响渠道**: Feishu
- **受影响桥接器**: Codex
- **受影响执行路径**: skill / prompt、agent shell transport、gateway internal run-tools、executor relay、outbound delivery、operator 查询面
- **运维影响**: media delivery 审计、传输接线可观测性、Feishu token 刷新审计、run 与 media 关联排障
- **范围外内容**: 跨 session / 跨用户发送、scheduler 主动投递、管理后台手工上传资源、任意渠道统一媒体抽象、任意二进制类型 SLA、把 agent 变成直接调用 Feishu API 的客户端

## 用户场景与测试 *(必填)*

### 用户故事 1 - 低心智负担地把资源发回当前会话（优先级：P1）

作为在飞书里与 agent 协作的使用者，我希望当我说“把截图发给我”或“把这个文件直接发出来”时，agent 能直接把资源发回当前聊天，而不是先花很多轮排查 PATH、worktree、`bun` 或 `runId`。

**优先级原因**: 当前实现已经证明底层链路可以跑通，但真正阻碍体验的是 agent 不会稳定、直接地使用这条能力。先修正主路径心智模型，功能才算可用。

**独立验证方式**: 在一个已经绑定 workspace 且存在活动 run 的飞书聊天里，请 agent 发送一个本地截图或文件。若 agent 直接走受支持的媒体发送路径，并且当前聊天收到资源本体，且中途不需要用户提供运行时上下文或让 agent 自行做环境考古，即视为通过。

**验收场景**:

1. **Given** 当前飞书 `chat` 中存在一个活动 run，且 agent 已能访问本地截图路径，**When** 用户说“把截图发送给我”，**Then** agent 应直接调用受支持的媒体发送能力，并把图片发回当前 `chat`，而不是要求用户补充 `runId`、`chatId`、`workspace` 或解释 PATH / worktree 问题。
2. **Given** agent 已生成一个本地文件，**When** 用户要求“直接发文件，不要只贴链接”，**Then** agent 在正常路径里只需提供资源业务参数，而不需要手工拼接 `runId`、`sessionId`、`chatId`、`gatewayBaseUrl` 或手动切换到特定 worktree 才能发送。
3. **Given** 当前媒体传输路径本身不可用，**When** agent 尝试发送资源，**Then** 系统必须快速返回明确失败，agent 应把它表述为“当前发送能力不可用”，而不是默认进入多轮环境排查。
4. **Given** `carvis-media send` 的首次正常尝试已经返回结构化失败，**When** agent 准备继续发送同一资源，**Then** 它必须停止并向用户报告失败，而不是继续搜索源码目录、切换 worktree、包装 `bun` 或手工补运行时上下文。

---

### 用户故事 2 - 接受多种资源来源（优先级：P2）

作为使用 agent 处理资料和生成结果的使用者，我希望它既能发送本地生成的文件，也能发送来自远端地址的资源，这样我不需要为了资源来源不同而切换工作方式。

**优先级原因**: 只支持单一来源会显著削弱该能力的实用性，但它建立在核心发送路径已经是可直接使用的前提上。

**独立验证方式**: 分别让 agent 发送一个本地产物和一个远端资源；若两者都能在当前飞书会话中按资源形式交付，即可独立验证。

**验收场景**:

1. **Given** agent 可访问一个本地文件路径，**When** 它请求发送该资源，**Then** 使用者在当前飞书 `chat` 中收到该文件。
2. **Given** agent 可访问一个远端资源地址，**When** 它请求发送该资源，**Then** 使用者在当前飞书 `chat` 中收到对应图片或文件，而不是仅收到原始地址文本。

---

### 用户故事 3 - 失败透明且会话隔离（优先级：P3）

作为依赖飞书会话上下文完成任务的使用者和运维者，我希望资源发送失败时有明确结果，并且任何资源都只能回到当前会话，这样既不会误发到别的会话，也便于定位失败原因。

**优先级原因**: 这部分决定能力是否可安全上线和可运维，但它建立在 agent 主路径已经稳定可调用的前提上。

**独立验证方式**: 模拟资源不可读、远端地址不可获取、上传失败、发送失败和无效上下文，并确认系统给出明确失败结果；同时验证 agent 不能把资源发往当前会话之外的目标。

**验收场景**:

1. **Given** 当前 run 已绑定一个飞书 `chat`，**When** agent 发起资源发送，**Then** 系统只允许把资源投递到该 `chat`，不允许改发到其他会话或用户。
2. **Given** 某次资源发送在获取、上传或发送阶段失败，**When** 失败发生，**Then** agent 获得明确失败结果，用户侧与运维侧都能看到与该次失败相匹配的可审计状态。
3. **Given** 当前聊天并不存在活动 run，**When** agent 触发资源发送，**Then** 系统必须立即拒绝，且 agent 不应把这类失败包装成“已经发出”。

### 边界与异常场景

- 当没有活动 run、run 已结束、或当前运行没有可用的 session 上下文时，资源发送必须立即失败，不能延迟投递，也不能降级为任意目标发送。
- 当资源来源是本地路径但文件不存在、不可读或超出宿主机当前权限时，系统返回明确失败结果，不得伪造“已发送”。
- 当资源来源是远端地址但无法获取、响应异常、内容不可用或后续处理失败时，系统返回明确失败结果，并保留失败阶段的审计信息。
- 当请求的资源类型当前渠道无法承载时，系统必须明确告知“不支持该资源类型”，而不是静默转成纯文本链接。
- 当同一工作区已经有一个活动运行时，现有 FIFO 排队与单活动运行约束保持不变；资源发送能力不得绕过或新增并发执行路径。
- 当执行器心跳或智能体进程丢失时，run 仍按既有 `heartbeat_expired` 或失败语义终结；未完成的资源发送不得在 run 终止后继续尝试。
- 除 Feishu access token 失效时允许沿用现有 token 刷新语义重试一次外，本功能 v1 不做额外自动重试；任一阶段首次失败即结束该次资源发送尝试并返回明确失败结果。
- 当一次资源发送失败但 run 其他文本输出已经成功产生时，系统必须把这次资源发送视为独立失败事件，而不是回写整个 run 为“完全成功”或“完全无输出”。
- 当媒体传输路径未正确注入当前 agent shell 环境时，系统必须把它视为 transport wiring 缺陷并给出明确诊断；正常产品路径不能依赖 agent 自行寻找本地包路径、切换 worktree 或显式调用 `bun` 来完成发送。
- 调试 flags、手工上下文注入和本地 shell 排障仅属于测试或运维兜底路径，不属于 agent 主路径契约。
- 对同一资源发送请求，agent 的正常主路径只允许一次 `carvis-media send` 尝试；第一次结构化失败后必须停止并报告，不得进入自主环境排查。

## 需求 *(必填)*

### 功能需求

- **FR-001**: System MUST 允许 agent 在活动 run 内触发面向当前会话的资源发送，而不要求用户额外发起命令或新 run。
- **FR-002**: System MUST 将该能力限定为当前 run 绑定会话的 session-scoped outbound capability，不允许 agent 指定任意 `chat`、`user` 或其他外部目标。
- **FR-003**: System MUST 支持至少两类资源来源：本地路径与远端地址。
- **FR-004**: System MUST 支持把图片和通用文件作为资源交付给当前飞书会话；若请求的资源类型不被当前渠道支持，系统必须返回明确的不支持结果。
- **FR-005**: System MUST 在缺少活动 run、缺少会话上下文、或 run 已终止时立即拒绝资源发送请求，并返回可判定的失败结果。
- **FR-006**: System MUST 为每次资源发送返回结构化结果，至少能区分成功、上下文无效、资源不存在或不可读、资源获取失败、渠道上传失败、最终发送失败和渠道不支持。
- **FR-007**: System MUST 允许 agent 在同一 run 内多次发送资源，并保证这些资源都回到同一个当前会话。
- **FR-008**: System MUST 将资源发送失败视为独立的能力失败，不得因为单次资源发送失败而篡改已产生的其他 run 输出或把失败伪装为成功。
- **FR-009**: System MUST 为每次资源发送尝试保留可审计记录，至少包含关联的 `run`、关联的会话、资源来源类别、结果状态和失败原因摘要。
- **FR-010**: System MUST 保持现有工作区单活动运行、FIFO 排队、取消、超时和心跳语义不变；资源发送能力不得引入绕过这些约束的执行通道。
- **FR-011**: System MUST 在资源已成功交付时，让飞书使用者在当前会话中直接看到对应资源本体，而不是仅看到“已发送”的文字说明或外链兜底。
- **FR-012**: System MUST 保留统一的能力语义，包括“仅向当前会话发送”“失败有结构化结果”“每次尝试可审计”，避免形成仅适用于飞书单一路径的特殊规则。
- **FR-013**: System MUST 以 skill 驱动的 agent-facing 能力契约暴露媒体发送，正常路径只要求 agent 提供资源业务参数，例如 `path`、`url`、`mediaKind`、`title`、`caption`。
- **FR-014**: System MUST NOT 要求 agent 在正常路径上理解或拼接 `runId`、`sessionId`、`chatId`、`workspace`、`gatewayBaseUrl`、`PATH`、`cwd`、`bun` 或特定 worktree 路径。
- **FR-015**: System MUST 由可信的 runtime 注入或 relay 恢复运行时上下文，并由 gateway 作为最终权威校验；agent 自带的目标标识或上下文标识不得成为授权依据。
- **FR-016**: System MUST 使当前受支持的 transport 在 agent 所处 shell 会话中可直接执行，不能要求 agent 先手动安装二进制、切换目录、包装 `bun run` 或搜索源码目录。
- **FR-017**: System MAY 保留 `carvis-media` 这类 shell transport 作为实现形式或调试入口，但其 debug flags、手工上下文参数和本地排障步骤 MUST 视为非主路径契约。
- **FR-018**: System MUST 让 skill、prompt、quickstart 和运维文档中的能力描述与真实运行时保证保持一致；如果某项上下文不能稳定自动解析，文档不得宣称它“已经自动解决”。
- **FR-019**: System MUST 将媒体发送 skill/prompt 约束为低自由度主流程：命中媒体交付意图时，agent 直接调用一次 `carvis-media send`，失败即停并向用户报告，不得在默认路径中继续做环境排查。
- **FR-020**: System SHOULD 在 skill/prompt 中提供少量高频正例，例如“把截图发给我”“把这个文件直接发出来”，以减少 agent 对媒体发送动作的二次推理。

### 运维与可观测性需求 *(执行流变化时必填)*

- **OR-001**: System MUST 定义并记录资源发送相关的 operator-visible states，至少包括请求接受、资源获取失败、渠道上传失败、发送失败、发送成功，以及 transport wiring 缺陷；若发生 Feishu access token 刷新，也必须可审计。
- **OR-002**: System MUST 明确记录每次资源发送与所属 run 的关联关系，以及取消、超时、run 终止和资源发送失败之间的行为边界。
- **OR-003**: System MUST 说明本功能不会改变既有锁、队列、heartbeat 和失败回收语义，并让运维侧能够从持久化状态或日志判断失败发生层级。
- **OR-004**: System MUST 让运维侧能够区分“资源发送链路本身失败”和“agent 没有正确学会使用发送能力”这两类问题，至少能从 skill/prompt 文案、日志和 media delivery 审计中定位。

### 关键实体 *(涉及数据时填写)*

- **Agent Media Send Request**: 表示 agent 基于 skill 决策发起的一次资源发送意图，只包含资源业务参数，不包含授权上下文。
- **Media Transport Context**: 表示由 bridge、executor relay 或受控 shell 注入恢复出的运行时上下文，包含 `run`、`session`、`chat`、`workspace` 等权威信息，不属于 agent 业务输入。
- **Session-scoped Media Send**: 表示一次由活动 run 发起、目标固定为当前会话的资源发送请求与结果。
- **Media Delivery Attempt**: 表示单次资源发送在获取、渠道上传、最终投递各阶段的状态与失败原因。
- **Run / Session**: 现有规范实体；本功能依赖其绑定关系来约束资源发送只能回到当前会话，并把每次发送纳入现有运行生命周期审计。

## 假设与依赖

- 当前版本的用户可见交付范围仅要求 Feishu 会话；其他渠道是否复用同一能力模型不属于本 spec 的交付范围。
- 远端资源是否可获取，取决于运行环境当时的网络可达性与目标资源可访问性；本功能保证明确结果，不保证所有远端地址都可用。
- 渠道对资源大小、格式或安全策略的限制继续生效；本功能要求这些限制在失败时以明确结果体现，而不是静默吞掉。
- 当前实现可以继续使用 shell transport，但用户认可的产品目标不是“agent 学会调 shell 环境”，而是“agent 稳定地把资源发回当前聊天”。
- 活动 run 仍然是安全边界与审计边界；本 spec 不放宽“只能在当前 session 内发送”的约束。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**: 在当前飞书会话内发起的资源发送请求中，至少 95% 的可访问图片或文件能够在单次 run 期间成功送达到同一会话。
- **SC-002**: 对于“把截图发给我”“把文件直接发出来”这类明确媒体交付请求，agent 在正常 happy path 下无需额外环境排查回合即可触发发送；正常路径不要求用户补充运行时标识。
- **SC-003**: 对于本地路径资源与远端地址资源，两类来源都各自具备可独立验证的成功交付路径，且使用者无需额外手工操作即可收到资源。
- **SC-004**: 100% 的资源发送请求都只能投递到发起该 run 的当前会话，不能跨会话、跨用户或跨工作区误发。
- **SC-005**: 对于获取失败、上传失败、发送失败、transport wiring 缺陷和渠道不支持五类主要失败，运维人员无需登录宿主机即可从持久化状态、日志或管理视图判断失败阶段。
- **SC-006**: 引入资源发送能力后，现有单活动运行、排队、取消和 heartbeat 相关行为不出现可观察语义回归。
- **SC-007**: 对“把截图发给我”“把这个文件直接发出来”这类高频请求，agent 在成功路径上不需要第二次尝试；在失败路径上不出现自主环境排查回合。
