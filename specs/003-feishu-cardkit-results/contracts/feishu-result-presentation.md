# 合同：Feishu 结果呈现

## 1. 过程卡片创建

### 触发条件

- 仅当某次运行进入 `run.started` 后，系统才允许创建运行中卡片。
- `run.queued` 不得触发过程卡片创建。

### 成功条件

- 为单次运行创建一张 Feishu `interactive` 卡片
- 卡片内容包含最小运行状态与可更新的输出区域
- 卡片 `config` 必须显式启用共享更新配置，确保后续可以通过消息更新接口持续修改同一张卡片
- 创建成功后，系统记录该卡片关联的 `message_id`、`card_id` 和主要输出区域标识

### 失败条件

- 若过程卡片创建失败，系统必须把该次呈现标记为 `degraded`
- 失败后不得再尝试维持过程卡片流式体验
- 若用户侧尚无任何成功送达的消息，系统必须改为发送单条终态富文本兜底消息

## 2. 过程卡片更新

### 输入契约

- 输入来自同一 `runId` 的有序 `agent.output.delta`
- `sequence` 必须单调递增
- `gateway` 负责合并与节流，不要求一条 delta 对应一次卡片更新

### 成功条件

- 更新后的卡片文本应接近原始 agent 输出
- 优先保留标题、列表、强调、代码块、路径、命令与结构化分段等富文本语义
- 卡片正文应优先使用飞书支持的结构化元素表达，例如 `div + lark_md`，而不是把完整 Markdown 文档直接作为单个未约束文本块投递
- 可包含最小必要的清洗、去重和截断
- 不得把尚未开始执行的排队请求伪装为正在输出

### 失败条件

- 一旦任意过程卡片更新失败，系统必须停止后续过程卡片更新
- 若该卡片此前已成功创建，系统必须优先把同一张卡片切换为终态，或至少留下可审计的终态增强失败记录
- 不得因为过程更新失败而额外发送第二条成功终态消息来破坏单消息语义

## 3. 完成态摘要卡

### 完成态要求

- 运行结束后，原过程卡片不得被删除
- 卡片必须切换为 `completed`、`failed` 或 `cancelled` 中的一种终态
- 终态卡片必须保留：
  - 最终状态
  - 简短摘要
  - 关键运行信息
  - 失败或取消时的简短原因
  - 最后一段可读输出
- 终态卡片必须在同一条消息中承载 agent 的终态结果内容，并优先保留 agent 自身的结构、语气与分段
- 当 agent 原始 Markdown 语法超出飞书卡片稳定支持范围时，系统必须将其归一化为等价的卡片分段或 `lark_md` 表达，例如将标题转换为分段标签而非依赖原始 `##` 语法
- 仅当终态内容明显缺少可读结构，或处于失败 / 取消 / 兜底路径时，系统才允许补充最小必要的状态、原因或下一步信息
- 正常成功路径不得再额外生成第二条终态结果消息

## 4. 终态兜底交付

### 成功条件

- 仅当过程卡片创建失败且用户侧尚无任何成功送达的消息时，系统才允许发送单条终态富文本兜底消息
- 该兜底消息至少区分：
  - 结论
  - 主要变更
  - 验证
  - 下一步

### 降级要求

- 即使过程卡片创建失败，用户最终仍必须收到一条终态交付
- 过程卡片是主交付路径；终态富文本兜底消息只属于异常降级路径
- 当过程卡片已成功创建时，系统不得再发送第二条成功终态消息
