# Research: Feishu 卡片结果增强

## 决策 1：采用“单张运行卡片演进为终态富文本卡片”的单消息呈现

- **Decision**: 运行开始后创建一张 Feishu `interactive` 过程卡片，并在运行结束后保留同一张卡片，切换为终态富文本摘要卡；只有在卡片创建失败且用户侧尚无任何成功交付时，才允许退化为单条终态富文本消息。
- **Rationale**: 单张消息能减少飞书会话噪声，并把过程与终态上下文保留在同一处；终态卡片本身承载结论、变更、验证和下一步，更符合“单消息交付”目标。
- **Alternatives considered**:
  - 运行中卡片 + 完成后额外 `post`: 实现更直接，但会让一次成功运行出现两条终态消息，破坏单消息体验。
  - 只发最终终态消息：无法满足“运行中可见”和打字机体验。

## 决策 2：排队中的请求不创建过程卡片

- **Decision**: 请求进入队列时不创建运行中卡片，仅保留现有排队可见性；只有在 `run.started` 后才创建过程卡片。
- **Rationale**: 这样不会制造“看起来在流式执行，但其实还没开始”的假象，也能保持现有队列语义稳定。
- **Alternatives considered**:
  - 入队即创建“排队中卡片”：用户体验更显眼，但会把未开始执行的请求误包装成正在工作。
  - 入队发阶段性文本提示：会引入额外文本噪声，和本轮“减少零散提示消息”的方向冲突。

## 决策 3：引入更细粒度的 `RunEvent` 输出，而不是继续只靠 `agent.summary`

- **Decision**: 在保持既有终态事件的同时，为 `bridge-codex` 增加面向呈现层的细粒度输出事件，例如 `agent.output.delta`，由 `gateway` 负责合并与节流后写入 CardKit。
- **Rationale**: 当前 `agent.summary` 粒度太粗，只适合阶段性摘要，不足以驱动“接近原始输出”的打字机体验。
- **Alternatives considered**:
  - 继续复用 `agent.summary`：实现简单，但无法达到接近原始输出的体验目标。
  - 直接让 `channel-feishu` 消费 Codex CLI 原始 stdout：会打破 `AgentBridge` 边界。

## 决策 4：新增 `RunPresentation` 作为持久化呈现实体

- **Decision**: 除了保留 `Run`、`RunEvent`、`OutboundDelivery` 外，新增一个面向单次运行呈现生命周期的 `RunPresentation` 实体，记录过程卡片、终态卡片、降级状态以及单消息约束是否被破坏。
- **Rationale**: 单消息卡片仍然包含创建、流式更新、终态切换和异常兜底等多个可独立失败的阶段；把它们压进 `Run` 或只靠 `OutboundDelivery` 都会损失生命周期语义。
- **Alternatives considered**:
  - 只扩展 `OutboundDelivery`: 交付记录有了，但缺少“同一轮运行的当前呈现状态”。
  - 把呈现字段直接塞进 `Run`: 会让运行实体混入过多渠道呈现细节。

## 决策 5：过程卡片更新失败后立即停止流式增强，但不放弃单消息终态

- **Decision**: 一旦过程卡片创建或更新失败，系统停止后续过程卡片流式更新并记录降级状态；若卡片已成功创建，则优先保住同一张卡片的终态切换；若卡片从未成功创建，则发送单条终态富文本兜底消息。
- **Rationale**: 主交付物已经变成单张卡片，不能因为局部更新失败就主动制造第二条成功消息；同时也不能让用户在卡片完全未送达时失去终态交付。
- **Alternatives considered**:
  - 高频重试卡片更新：可能把局部失败放大成整轮投递抖动。
  - 失败后改发过程文本摘要：会把群聊重新拉回文本刷屏。

## 决策 6：终态摘要卡承载结构化富文本结果，并保留最后一段可读输出

- **Decision**: 完成态摘要卡保留终态、结构化结论、主要变更、验证、下一步、关键运行信息和最后一段可读输出；不再依赖额外 `post` 承载完整终态结果。
- **Rationale**: 这样既能保留执行过程的上下文，又能让同一条终态卡片本身成为完整交付物，而不会把关键信息拆散到第二条消息里。
- **Alternatives considered**:
  - 完整保留全部过程输出：卡片会过长，且阅读负担大。
  - 只保留状态和原因：回看时缺少最后的上下文线索。

## 决策 7：在 `gateway` 做输出窗口合并、节流和有限富文本恢复

- **Decision**: `gateway` 基于细粒度输出事件维护一个滑动输出窗口，并以节流后的更新节奏推动过程卡片，同时尽量恢复标题、列表、强调、代码块、路径和命令等富文本语义，而不是把所有内容拍平成纯文本。
- **Rationale**: 这样既保留“接近原始输出”的感受，又避免高频更新带来的噪声和外部接口压力，并减少终态和流式阶段的富文本丢失。
- **Alternatives considered**:
  - 每个 delta 直接更新卡片：实现直观，但很容易产生频控和阅读噪声问题。
  - 只按固定时间轮询聚合：会让呈现层和真实运行事件脱钩过深。
  - 全量保留原始 Markdown/富文本 AST: 保真度更高，但当前阶段复杂度和跨边界成本过高。
