# 快速验证：Feishu 卡片结果增强

## 1. 前置条件

确保 `002-local-runtime-wiring` 的本地双进程联调已经成立：

- `~/.carvis/config.json` 可用
- `POSTGRES_URL`、`REDIS_URL`、`FEISHU_APP_ID`、`FEISHU_APP_SECRET` 已配置
- `gateway` 与 `executor` 可分别启动
- Feishu websocket 接入正常
- `codex` CLI 可运行

## 2. 启动本地 runtime

- 终端 A: `bun run start:gateway`
- 终端 B: `bun run start:executor`
- 可选联调开关：
  - `CARVIS_FAIL_CARD_CREATE=1 bun run start:executor`
  - `CARVIS_FAIL_CARD_UPDATE=1 bun run start:executor`
  用于人工验证降级路径。

## 3. 验证运行中卡片

在飞书里发送一个预期会产生较多中间输出的请求，例如：

```text
请检查当前仓库并总结架构、最近改动和测试状态
```

预期结果：

- 用户原消息先出现工作中 reaction
- 真正开始执行后，出现一张运行中 `interactive` 卡片
- 过程卡片的主要输出区域持续更新，内容接近原始 agent 输出

## 4. 验证排队语义不变

在第一条请求仍在运行时，再发送第二条普通请求。

预期结果：

- 第二条请求不会立即创建过程卡片
- 只有在它真正开始执行后，才创建对应的运行中卡片

## 5. 验证完成态摘要卡为单条终态交付

等待运行完成。

预期结果：

- 运行中卡片停止更新并保留
- 卡片切换为完成态摘要卡
- 不会再额外发送第二条成功终态消息
- 同一张卡片终态优先保留 agent 原始回答结构，不会被强制改写为固定四段式模板

## 6. 验证失败 / 取消终态

至少验证一个异常路径：

- 发送 `/abort`
- 或触发一个可控失败场景

预期结果：

- 过程卡片切换为失败态或取消态摘要卡
- 卡片保留最后一段可读输出和简短原因
- 若过程卡片此前已成功送达，则不会额外发送第二条成功终态消息

## 7. 验证降级行为

人为模拟过程卡片创建失败或更新失败。推荐直接使用环境变量：

```bash
CARVIS_FAIL_CARD_UPDATE=1 bun run start:executor
```

预期结果：

- 若为卡片更新失败，系统停止继续流式更新过程卡片
- 不再刷新的过程卡片不会被频繁重试
- 若卡片创建从未成功，则系统发送一条终态富文本兜底消息
- 若卡片已经成功创建，则系统不得再额外发送第二条成功终态消息
- 运维侧可从持久化状态或日志判断发生过降级

## 8. 自动化验证范围

本轮任务完成后，自动化测试至少应覆盖：

- `bridge-codex` 输出事件从 `agent.summary` 扩展到更细粒度输出事件
- 运行真正开始后才创建过程卡片
- 过程卡片在终态切换为完成态摘要卡
- 正常路径满足“一次运行一条用户可见成功消息”
- 流式和终态阶段尽可能保留标题、列表、强调、代码块、路径或命令等富文本语义
- 卡片创建失败后使用单条终态富文本兜底，卡片更新失败后不再补发第二条成功终态消息
- 群聊与单聊保持一致的 reaction / card 语义
