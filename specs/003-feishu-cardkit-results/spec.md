## Clarifications

### Session 2026-03-08

- Q: 排队中的请求应何时创建运行中卡片？ → A: 请求进入队列时不创建卡片，真正开始执行后才创建运行中卡片。
- Q: 最终结果应采用哪种消息形式？ → A: 该轮早期讨论曾倾向单独 `post`，但已被 2026-03-09 的澄清覆盖；当前以同一张终态富文本卡片为主，只有卡片创建失败时才允许单条终态富文本兜底消息。
- Q: 失败或取消时的完成态摘要卡应保留哪些内容？ → A: 保留最后一段可读输出，并显示失败或取消状态与简短原因。
- Q: 运行中卡片更新失败后应如何降级？ → A: 停止过程卡片更新并优先保住同一张卡片的终态切换；只有卡片从未成功创建时，才允许退化为单条终态富文本兜底消息。

### Session 2026-03-09

- Q: 一次成功运行是否允许同时出现运行卡片和独立最终结果消息？ → A: 不允许。正常路径必须收敛为单条消息，由同一张卡片从运行中演进到终态。
- Q: 最终交付应采用哪种消息形式？ → A: 以同一张 Feishu `interactive` 卡片承载终态富文本结果；只有在卡片创建失败导致用户侧尚无任何交付物时，才允许退化为单条终态富文本消息。
- Q: 富文本保真目标应如何定义？ → A: 优先保留标题、列表、强调、代码块、路径、命令与结构化分段；不要求逐 token 原样还原，但不得把本可表达为结构化富文本的内容提前拍平成单段纯文本。

### 用户故事 1 - 运行中卡片输出（优先级：P1）

飞书用户发起普通请求后，希望立即看到一张运行中卡片，并在卡片中持续看到接近原始 agent 输出的文本更新，而不是只看到静态等待状态或零散提示消息。

**优先级原因**: 这是本功能的核心价值。如果没有运行中卡片与持续输出，用户仍然无法明显感知系统正在做什么，也无法获得比当前文本提示更好的执行体验。

**独立验证方式**: 在飞书会话中发起一次普通请求，确认系统发送运行中卡片，并在执行过程中持续更新卡片内容即可完整验证。

**验收场景**:

1. **Given** 用户在飞书会话中发送普通请求且系统成功受理，**When** 运行开始执行，**Then** 系统会发送一张运行中卡片，并在卡片中持续更新接近原始 agent 输出的文本内容。
2. **Given** 运行中卡片已创建，**When** agent 输出出现重复或超长片段，**Then** 系统会对输出做最小必要清洗与截断，确保卡片仍然可读。
3. **Given** 一个请求已经进入队列但尚未开始执行，**When** 用户等待该请求被调度，**Then** 系统不会提前创建运行中卡片，而是仅保留既有排队可见性方式，直到该请求真正开始执行。

---

### 用户故事 2 - 保留完成态摘要卡（优先级：P2）

飞书用户在任务完成、失败或取消后，希望原有运行中卡片被保留下来并切换为完成态摘要卡，这样可以回看本轮执行过程和最终状态，而不是丢失过程信息。

**优先级原因**: 这直接决定运行卡片是否只是临时噪声，还是可回看、有上下文价值的交付物。

**独立验证方式**: 发起一次普通请求并等待其完成、失败或取消，确认运行中卡片停止更新并保留为完成态摘要卡即可验证。

**验收场景**:

1. **Given** 一次普通请求已经存在运行中卡片，**When** 运行完成，**Then** 该卡片停止打字机更新并切换为展示最终状态、摘要和关键运行信息的完成态摘要卡。
2. **Given** 一次普通请求已经存在运行中卡片，**When** 运行失败或被取消，**Then** 该卡片保留并切换为失败态或取消态摘要卡，明确显示结果状态、简短原因以及最后一段可读输出。

---

### 用户故事 3 - 单消息富文本终态（优先级：P3）

飞书用户在任务结束后，希望仍只看到同一条消息，但这条消息已经演进为包含结论、变更、验证和下一步的富文本终态卡片，而不是再收到第二条结果消息，或看到被拍平成普通文本的终态内容。

**优先级原因**: 这直接决定最终体验是否简洁、可回看且信息层次清晰，也是“单消息交付”是否真正成立的关键。

**独立验证方式**: 发起一次普通请求并等待结束，确认系统不再额外发送第二条最终结果消息，且原卡片终态本身已承载结构化富文本结果即可验证。

**验收场景**:

1. **Given** 一次普通请求已经结束且过程卡片创建成功，**When** 系统生成终态结果，**Then** 用户只保留同一张卡片消息，卡片终态应优先保留 agent 自身的原始结构、语气与分段，不会再收到第二条成功结果消息。
2. **Given** 运行过程中存在列表、强调、代码块、路径或命令等可结构化内容，**When** 系统更新运行中卡片或切换终态卡片，**Then** 系统应尽可能保留这些富文本语义，而不是统一退化为单段普通文本。
3. **Given** 过程卡片创建失败且用户侧尚无任何成功送达的消息，**When** 运行结束，**Then** 系统允许退化为单条终态富文本消息以保证交付，但同一轮运行仍不得同时保留成功卡片和独立最终结果两条成功消息。

### 边界与异常场景

<!--
  必填：请填写真实的边界条件和异常场景，不要保留占位内容。
-->

- 当一次运行没有产生足够多的中间输出时，系统仍应发送运行中卡片，但允许只显示简短的进行中状态与少量最新输出。
- 当中间输出过长、重复或包含明显无助于阅读的噪声时，系统应进行最小必要清洗、合并或截断，但不得改变用户对运行状态的基本判断。
- 当工作区中已经有一个活动运行时，新请求仍按既有队列语义处理；如果该请求尚未开始执行，则不得提前创建运行中卡片或生成伪造的进行中输出内容。
- 当执行器心跳丢失、智能体进程异常退出或运行超时时，运行中卡片必须切换为失败态摘要卡，并在同一张卡片中明确说明失败原因或取消原因。
- 当卡片创建失败、卡片更新失败或后续更新频率受限时，系统必须优先维持“同一张卡片继续存在”的单消息体验；仅当卡片创建失败导致用户侧尚无任何成功消息时，才允许降级为单条终态富文本消息，并在运维侧留下可见失败记录。
- 当运行中卡片更新已经失败但卡片本身已成功创建时，系统不得额外发送第二条成功终态消息来补齐结果，而应在可能范围内将原卡片切换到终态，或在无法更新时明确记录“已有单消息但终态增强失败”的运维状态。
- 当出站投递重试耗尽时，用户侧应至少保留一条已成功送达的终态交付物；运维侧必须能看到该次投递失败的原因和重试结果。

## 需求 *(必填)*

### 功能需求

- **FR-001**: System MUST 在飞书普通请求真正进入执行后，为该次运行发送一张独立的运行中卡片；仅进入队列而尚未开始执行的请求不得提前创建运行中卡片。
- **FR-002**: System MUST 在运行过程中持续更新运行中卡片的主要输出区域，使其尽可能接近原始 agent 输出，并形成连续可感知的打字机式变化；对可结构化内容应优先保留富文本语义而不是统一拍平成单段普通文本。
- **FR-003**: System MUST 对运行中输出执行最小必要的清洗、合并与截断，避免明显重复、噪声或超长内容破坏卡片可读性。
- **FR-004**: System MUST 在运行结束时保留原有运行中卡片，并将其切换为完成态摘要卡，而不是删除或替换为纯状态提示。
- **FR-005**: System MUST 在完成态摘要卡中展示本轮运行的最终状态、结构化结论以及对用户有判断价值的关键运行信息；当运行失败或取消时，摘要卡还必须保留最后一段可读输出和简短原因。
- **FR-006**: System MUST 让同一张 Feishu 卡片在终态中直接承载 agent 的终态结果内容，并优先保留 agent 自身的结构、语气与分段；仅在内容明显缺少可读结构或进入异常兜底路径时，才允许做最小必要的结构化补全；不得在正常成功路径中额外发送第二条最终结果消息。
- **FR-007**: System MUST 保持现有 `/status`、`/abort` 与普通消息运行语义不变，本功能只增强普通请求的结果呈现方式。
- **FR-008**: System MUST 在卡片创建失败、卡片更新失败或卡片更新受限时自动降级，并保证用户最终仍能收到一条终态交付；在卡片已成功创建的前提下，系统不得再通过额外成功消息破坏单消息语义。
- **FR-009**: System MUST 使同一次运行的流式卡片内容、终态卡片内容与降级交付能够被统一关联，以便用户和运维定位这些呈现均属于同一轮执行。
- **FR-010**: System MUST 在群聊与单聊中保持一致的结果呈现语义，除非飞书会话能力限制要求明确降级。
- **FR-011**: Users MUST be able to 从单条终态交付中快速识别本轮是否成功，并在 agent 已明确给出时读到修改、验证或下一步等关键信息；呈现层不得为追求固定模板而覆盖 agent 原始表达。
- **FR-012**: System MUST 在正常成功路径中保证“一次运行对应一条用户可见成功消息”；同一轮运行若产生第二条用户可见终态消息，必须被视为契约违背而非增强体验。

### 运维与可观测性需求 *(执行流变化时必填)*

- **OR-001**: System MUST 为运行中卡片创建、卡片更新、完成态切换、终态富文本交付与降级发送记录明确的 operator-visible 状态和结构化日志。
- **OR-002**: System MUST 明确定义卡片创建失败、卡片更新失败、频控限制、运行失败、取消、超时和投递重试耗尽时的用户可见行为与运维可见行为；其中卡片更新失败后的标准行为是停止过程卡片流式增强并优先保住单消息终态语义，而不新增独立 CardKit 重试状态机。
- **OR-003**: System MUST 保持现有锁、队列、心跳与执行路径语义不变，并记录本功能只改变出站呈现层而不改变工作区串行化规则。
- **OR-004**: System MUST 让运维能够从持久化状态或日志中判断某次运行是否成功创建过程卡片、是否发生降级、是否违反单消息约束，以及终态交付是否成功送达。

### 关键实体 *(涉及数据时填写)*

- **Run Presentation**: 表示某次运行在飞书侧的呈现状态，包含流式卡片状态、终态卡片状态、降级交付状态以及是否违反单消息约束。
- **Streaming Card View**: 表示运行中卡片的用户可见内容快照，包含当前展示文本、富文本片段、最后更新时间和是否处于活跃更新状态。
- **Terminal Rich Card View**: 表示同一张卡片在终态时展示的富文本内容，优先保留 agent 原始结构，并在必要时补充最小状态摘要或兜底信息。
- **Run / Delivery / Session**: 沿用既有规范实体，并在本功能范围内增加“同一轮运行正常情况下只对应一条用户可见成功消息”这一交付语义。

## 假设

- 飞书会话具备发送和更新可承载富文本结构的 `interactive` 卡片能力。
- 运行中卡片的打字机体验以用户可感知的增量更新为准，不要求逐 token 原样转发。
- 终态富文本结果以中文为主，路径、命令、代码标识和结构化 ID 保持原文。
- 当卡片创建在用户侧完全失败时，允许使用单条终态富文本消息作为兜底交付，但该消息属于异常降级路径而非常规成功路径。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**: 在普通请求成功进入执行的场景中，95% 的运行会在用户首次可见的执行反馈出现后 5 秒内展示运行中卡片。
- **SC-002**: 在有中间输出的运行中，90% 的运行会至少向运行中卡片写入 3 次用户可感知的增量更新，且这些更新能保留标题、列表、强调、代码块、路径或命令中的至少一种结构化富文本语义。
- **SC-003**: 100% 的运行结束后都会留下可回看的完成态摘要卡，明确显示成功、失败或取消中的一种最终状态。
- **SC-004**: 在卡片创建成功的正常路径中，95% 的运行结束后不会再额外产生第二条成功终态消息，且用户可以直接在原卡片终态中读到 agent 的原始终态回答而非被强制改写为固定模板。
- **SC-005**: 当过程卡片创建或更新失败时，运维无需登录宿主机即可从持久化状态或日志判断发生了降级，并确认用户最终是否至少收到一条终态交付。
