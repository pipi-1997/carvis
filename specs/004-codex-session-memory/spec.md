# 功能规格说明：Codex 会话续聊记忆

**功能分支**: `004-codex-session-memory`
**创建日期**: 2026-03-09
**状态**: 草稿
**输入**: 用户描述："为 carvis 增加基于 Codex 原生 session 的会话续聊记忆：同一飞书 chat 绑定并续用同一个 Codex session，支持 session 失效后自动重建、显式 /new 重置，以及 /status 暴露当前续聊状态；首版不做 MEMORY.md、长期记忆、历史摘要注入、scheduler 或主动 heartbeat。"

## 系统影响 *(必填)*

- **受影响渠道**: Feishu
- **受影响桥接器**: Codex
- **受影响执行路径**: gateway ingress, queueing, executor, outbound delivery
- **运维影响**: queueing, retries, notifications, admin visibility
- **范围外内容**: `MEMORY.md`、长期记忆、历史摘要注入、scheduler、主动 heartbeat、多 agent / 多 workspace 记忆共享

## 用户场景与测试 *(必填)*

### 用户故事 1 - 同会话续聊（优先级：P1）

作为在飞书里连续和机器人的使用者，我希望同一个 `chat` 中后续请求能够延续前一轮的上下文，而不是每次都从零开始，这样我可以围绕同一个任务持续追问、补充指令和修正方向。

**优先级原因**: 这是本功能的核心用户价值，也是当前体验中最明显的能力缺口。

**独立验证方式**: 在同一个飞书 `chat` 中连续发送两条有明显上下文依赖的请求，第二条请求无需重复背景即可获得基于前文的有效回答。

**验收场景**:

1. **Given** 某个飞书 `chat` 首次发起运行且此前没有续聊绑定，**When** 该运行完成，**Then** 系统为该 `chat` 保存可供后续续聊使用的 Codex session 绑定。
2. **Given** 某个飞书 `chat` 已有有效的 Codex session 绑定，**When** 用户在同一 `chat` 发起下一轮请求，**Then** 系统使用该绑定继续对话，而不是从全新上下文开始。

---

### 用户故事 2 - 显式开新话题（优先级：P2）

作为在同一个飞书 `chat` 中经常切换任务的使用者，我希望能够显式清空当前续聊上下文并从新会话开始，这样新话题不会被旧上下文污染。

**优先级原因**: 没有重置入口会让续聊能力难以可靠使用，也会降低用户对“记忆边界”的控制感。

**独立验证方式**: 在已有续聊绑定的飞书 `chat` 中执行重置命令后，再发送一条依赖旧上下文的问题，应表现为新会话而非继续旧上下文。

**验收场景**:

1. **Given** 当前飞书 `chat` 已绑定续聊 session，**When** 用户发送 `/new`，**Then** 系统清空该 `chat` 的续聊绑定并明确告知后续请求将从新会话开始。
2. **Given** 用户刚执行 `/new`，**When** 用户发送下一条普通消息，**Then** 系统为其创建新的续聊会话，而不是恢复旧绑定。

---

### 用户故事 3 - 失效后自动恢复（优先级：P3）

作为依赖续聊上下文的使用者，我希望当底层 Codex session 已失效时，系统能自动恢复到一个新的可用会话，而不是把恢复负担完全留给我，这样我仍能继续完成任务。

**优先级原因**: 失效恢复不是首要价值，但如果没有这一层，续聊一旦失效就会变成脆弱能力。

**独立验证方式**: 模拟一个已保存但不可恢复的 Codex session，再在同一 `chat` 中发起请求，系统应自动重建会话并继续完成该请求。

**验收场景**:

1. **Given** 飞书 `chat` 已保存一个不可恢复的续聊 session，**When** 用户发送新的普通消息，**Then** 系统先识别该续聊绑定失效，再自动重试一次新会话并完成请求。
2. **Given** 续聊绑定失效且自动重试成功，**When** 用户再次发送后续消息，**Then** 系统继续使用新绑定，而不是反复尝试旧失效绑定。

### 边界与异常场景

- 当某个 `chat` 从未建立过续聊绑定时，系统按新会话执行，不应向用户表现为错误。
- 当用户在同一个 `workspace` 中排队等待执行时，排队语义保持不变；续聊绑定不得绕过现有 FIFO 队列与单活动运行约束。
- 当多个飞书 `chat` 共享同一个 `workspace` 时，每个 `chat` 的续聊绑定必须彼此隔离，不得因为共享队列或共享工作区而串用上下文。
- 当底层续聊 session 因过期、丢失或不可恢复而失效时，系统只自动重试一次新会话；若仍失败，则按正常运行失败回传。
- 当失败原因并非“续聊 session 无效”而是普通执行失败时，系统不得擅自丢弃现有续聊绑定。
- 当执行器心跳或智能体进程丢失时，run 仍按既有 `heartbeat_expired` / 失败语义处理；本功能不得改变现有锁释放、取消和失败归因规则。
- 当出站投递重试耗尽时，用户侧仍按现有投递失败表现处理；运维侧必须能区分“续聊绑定已更新但通知失败”和“运行本身失败”。
- 当用户执行 `/new` 时，当前活动运行不被中断；命令只影响之后的新请求。

## 需求 *(必填)*

### 功能需求

- **FR-001**: System MUST 为每个飞书 `chat` 维护一个可选的 Codex 续聊绑定，用于表示该会话当前应延续的底层对话上下文。
- **FR-002**: System MUST 在首轮无绑定请求成功结束后，为当前 `chat` 建立或更新该续聊绑定，使后续请求可继续同一对话上下文。
- **FR-003**: System MUST 在同一飞书 `chat` 的后续普通消息触发运行时，默认尝试使用当前续聊绑定继续对话。
- **FR-004**: Users MUST be able to 通过 `/new` 清空当前飞书 `chat` 的续聊绑定，并让后续普通消息从新会话开始。
- **FR-005**: Users MUST be able to 通过 `/status` 查看当前飞书 `chat` 是否已绑定续聊上下文，以及该绑定当前处于“新会话模式”“续聊模式”“最近显式重置后等待新会话”或“最近自动恢复后继续会话”；若最近一次自动恢复失败，也必须给出明确提示。
- **FR-006**: System MUST 在检测到底层续聊 session 不可恢复时，自动清理旧绑定并只重试一次新会话执行。
- **FR-007**: System MUST 在自动重试成功后，把新的续聊绑定回写到当前飞书 `chat`，供后续请求继续使用。
- **FR-008**: System MUST 在自动重试仍失败时，按普通运行失败返回明确结果，并避免把失败伪装成已恢复成功。
- **FR-009**: System MUST 保持现有 `workspace` 单活动运行、FIFO 排队、取消、超时和心跳语义不变；续聊能力不得绕过这些约束。
- **FR-010**: System MUST 让使用者能够明确区分“继续当前上下文”和“开始全新上下文”这两种行为，而不要求其重复输入底层技术细节。

### 运维与可观测性需求 *(执行流变化时必填)*

- **OR-001**: System MUST 定义并记录续聊绑定相关的 operator-visible states，包括“未绑定”“已绑定”“显式重置”“绑定失效后自动重建”“自动重建失败”。
- **OR-002**: System MUST 明确记录续聊失效判定、自动重试是否发生、最终是否回写新绑定，以及失败后沿用的通知与投递行为。
- **OR-003**: System MUST 说明本功能不会改变现有锁、队列、run heartbeat 与失败回收规则，并让运维侧可以区分“续聊恢复问题”与“普通 run 失败”。

### 关键实体 *(涉及数据时填写)*

- **Conversation Session Binding**: 表示某个飞书 `chat` 当前绑定的底层 Codex 续聊上下文，包含绑定状态、最后一次成功确认时间和最近一次失效/重置结果。
- **Session**: 现有飞书 `chat` 到固定 agent / workspace 的绑定记录；本功能在其语义范围内补充续聊上下文状态。
- **Run**: 单次执行请求；运行本身仍然独立排队、执行和终结，但可以引用当前会话的续聊绑定。
- **Run Event**: 表示 run 生命周期中的规范事件；本功能会引入续聊绑定建立、失效恢复或重置相关的事件语义或等价的可审计状态。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**: 在同一个飞书 `chat` 中，至少 90% 的连续上下文追问能够在不重复背景的情况下得到正确延续的回答。
- **SC-002**: 当底层续聊 session 已失效时，系统能在单次用户请求内自动恢复到可用新会话，并在一次自动重试后给出明确成功或失败结果。
- **SC-003**: 使用者在执行 `/new` 后，下一条请求能够稳定表现为新上下文，而不是错误续用旧上下文。
- **SC-004**: 运维人员无需登录宿主机，即可从持久化状态、日志或管理视图判断某个请求是“正常续聊”“显式重置后新会话”还是“失效后自动重建”。
- **SC-005**: 引入续聊记忆后，现有工作区串行化、排队、取消和心跳失效相关行为不出现可观察语义回归。
