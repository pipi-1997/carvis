# Research: Codex 会话续聊记忆

## 决策 1：采用 Codex 原生 session 续聊，而不是历史摘要回灌

- **Decision**: 首版直接复用底层 Codex 原生 session 作为同一飞书 `chat` 的续聊上下文，不通过拼接历史 `RunEvent` 或摘要文本来模拟记忆。
- **Rationale**: 这样最接近用户对“同一会话继续聊”的直觉，也能避免摘要回灌带来的信息丢失、token 浪费和多层压缩失真。
- **Alternatives considered**:
  - 仅靠最近几轮 run 摘要回灌：实现上不依赖底层 session，但记忆保真度差。
  - 同时做原生 session + 摘要回退：更稳，但首版复杂度过高。

## 决策 2：将续聊绑定建模为独立持久化实体，而不是直接塞进 `Session`

- **Decision**: 新增独立的 `ConversationSessionBinding` 持久化实体，表达飞书 `chat` 当前的底层 Codex session 绑定及其状态；`Session` 继续表示渠道会话和 workspace 绑定。
- **Rationale**: 这样可以把“渠道会话路由”与“底层续聊上下文”分开，既便于记录重置、失效和自动重建状态，也避免把 `Session` 膨胀成过多桥接器细节。
- **Alternatives considered**:
  - 直接在 `sessions` 表新增 `codex_session_id`：实现更快，但不利于记录失效原因、重置时间和自动恢复结果。
  - 把续聊状态挂到 `Run`：会把短生命周期运行实体与长生命周期会话上下文耦合在一起。

## 决策 3：自动恢复只做一次新会话重试

- **Decision**: 当系统能明确判定当前续聊 session 不可恢复时，只允许自动清理旧绑定并重试一次新会话执行。
- **Rationale**: 一次重试足以覆盖常见的 session 过期或缺失场景，同时能防止无限重试、重复副作用和对队列/工作区锁造成额外干扰。
- **Alternatives considered**:
  - 不自动恢复，直接失败：用户需要手动 `/new`，体验过于脆弱。
  - 多次自动重试：恢复率可能更高，但会放大外部不稳定性并增加副作用风险。

## 决策 4：仅把“可识别的续聊无效”作为自动恢复触发条件

- **Decision**: 只有当底层错误可明确识别为“续聊 session 不存在、已过期或不可恢复”时，系统才清空绑定并自动重试；普通 bridge 执行失败不触发绑定清理。
- **Rationale**: 这样可以避免把临时网络波动、命令失败或工作区问题误判为续聊上下文损坏。
- **Alternatives considered**:
  - 任何 bridge 错误都清空绑定：简单，但会错误丢弃仍然有效的上下文。
  - 从不清理绑定：会导致系统反复尝试已失效的 session。

## 决策 5：显式重置命令使用 `/new`

- **Decision**: 首版通过 `/new` 提供显式开新会话能力，命令只影响后续新请求，不中断当前 active run。
- **Rationale**: `/new` 语义直观，也和同类对话产品中“开始新会话”的用户心智接近；不影响当前运行可以避免把命令重置和运行取消混在一起。
- **Alternatives considered**:
  - 复用 `/abort`：会混淆“终止当前运行”和“重置后续上下文”。
  - 使用更底层的 `/reset-session`：更精确，但用户学习成本更高。

## 决策 6：`/status` 只暴露轻量续聊状态，不暴露底层标识细节

- **Decision**: `/status` 首版只需告诉用户当前是“新会话模式”还是“续聊模式”，以及是否刚发生过显式重置或自动重建；不要求直接暴露完整底层 session 标识。
- **Rationale**: 用户需要的是行为状态和可预期性，而不是底层实现细节；运维和开发的详细定位仍由持久化状态和结构化日志承担。
- **Alternatives considered**:
  - 完整输出底层 session id：便于调试，但用户价值低且增加实现细节泄漏。
  - `/status` 不展示续聊状态：会让用户无法判断当前上下文边界。

## 决策 7：首版不引入长期记忆或摘要记忆回退

- **Decision**: `004` 只覆盖会话续聊记忆，不在本轮引入 `MEMORY.md`、长期知识沉淀、历史摘要注入或 memory search。
- **Rationale**: 当前最直接的缺口是“看似同一飞书会话，底层却每轮失忆”；先补齐这一层，后续再考虑长期记忆和上下文压缩更可控。
- **Alternatives considered**:
  - 同时接入长期记忆：功能更完整，但边界和写入策略复杂度高。
  - 在续聊失效时自动回退到摘要记忆：稳健性更好，但会在首版引入第二种记忆来源和额外的不确定性。
