# 功能规格说明：飞书 Codex 对话闭环

**功能分支**: `001-feishu-codex-mvp`
**创建日期**: 2026-03-08
**状态**: 草稿
**输入**: 用户描述："实现一个基于 Bun 的 Feishu + Codex 对话执行闭环：同一个机器人服务多个飞书会话，所有会话路由到同一个本地配置声明的 Codex agent；该 agent 使用固定 workspace 执行请求，支持普通消息发起运行、/status 查看状态、/abort 取消运行，并具备队列、锁、失败回传与持久化状态。"

## 系统影响 *(必填)*

- **受影响渠道**: Feishu
- **受影响桥接器**: Codex
- **受影响执行路径**: gateway ingress, queueing, executor, outbound delivery
- **运维影响**: locks, queueing, retries, notifications, admin visibility
- **范围外内容**: Telegram、Claude Code、多 agent 管理、运行时切换 workspace、scheduler、external webhook、admin UI

## 用户场景与测试 *(必填)*

### 用户故事 1 - 对话触发运行（优先级：P1）

作为在飞书里与机器人的使用者，我希望直接发送自然语言消息就能触发一次 Codex 运行，并在同一会话中持续看到运行状态、阶段性摘要与最终结果，这样我不需要切换到终端或管理后台也能完成一次完整的 AI 协作。

**优先级原因**: 这是产品的核心价值闭环；如果这个场景不成立，其他状态查询或取消能力都没有独立价值。

**独立验证方式**: 在一个已接入机器人的飞书会话中发送普通消息，系统自动绑定到本地配置声明的默认 agent 与 workspace，并返回排队、开始执行、阶段性摘要和最终完成或失败结果。

**验收场景**:

1. **Given** 机器人已接入飞书且本地 agent 配置完整，**When** 用户首次在某个飞书会话发送普通消息，**Then** 系统为该会话创建或解析 session、绑定默认 agent 与默认 workspace，并启动一次运行。
2. **Given** 某次运行已经开始，**When** Codex 持续产生执行结果，**Then** 系统将规范化后的状态变化与阶段性摘要回推到同一飞书会话，并在结束时返回最终结果。
3. **Given** 同一个飞书 `chat` 中存在多条连续消息，**When** 系统解析会话上下文，**Then** 这些消息都归属于同一个 session，而不是按 topic 或用户再次拆分。

---

### 用户故事 2 - 查询当前状态（优先级：P2）

作为飞书中的使用者，我希望通过 `/status` 查看当前会话对应 agent 的 workspace、当前运行状态，以及与当前会话最近一次请求相关的排队信息，这样我可以在不重复发起请求的情况下判断系统是否正在工作。

**优先级原因**: 当运行是异步且可能排队时，状态查询是降低不确定性的最小配套能力。

**独立验证方式**: 在飞书会话里发送 `/status`，系统返回当前 session 绑定信息、是否有 active run、最近一次运行状态，以及“当前会话最近一次请求是否处于排队中”和“若处于排队中则前方还有多少项”。

**验收场景**:

1. **Given** 当前 agent 没有活动运行，**When** 用户发送 `/status`，**Then** 系统返回绑定的 agent、workspace 和“当前无活动运行”的状态说明。
2. **Given** 当前 agent 存在活动运行或排队请求，**When** 用户发送 `/status`，**Then** 系统返回当前运行状态，以及当前会话最近一次请求是否排队和其前方队列长度；系统不返回完整队列列表。

---

### 用户故事 3 - 取消当前运行（优先级：P3）

作为飞书中的使用者，我希望通过 `/abort` 取消当前正在执行的运行，这样当请求明显不再需要或输出异常时，我可以主动终止资源占用。

**优先级原因**: 取消能力属于运行闭环的必要兜底，但其价值建立在前两个故事已经可用的前提上。

**独立验证方式**: 在存在 active run 的飞书会话中发送 `/abort`，系统取消当前运行并回传取消结果；若当前无 active run，则返回明确提示。

**验收场景**:

1. **Given** 当前 agent 有一个活动运行，**When** 用户发送 `/abort`，**Then** 系统只取消该活动运行，并将结果回传到会话。
2. **Given** 当前 agent 没有活动运行，**When** 用户发送 `/abort`，**Then** 系统返回可理解提示且不影响排队中的其他请求。

### 边界与异常场景

- 当本地 agent 配置缺失 `bridge` 或 `workspace` 时，系统必须拒绝发起运行，并向飞书会话返回可理解错误。
- 当同一 workspace 已经存在一个活动运行时，新请求必须进入 FIFO 队列，并明确提示用户当前请求已排队。
- 当执行器心跳丢失、Codex 进程异常退出或运行超时时，系统必须把运行标记为失败或取消，并保持状态可查询。
- 当出站投递多次重试后仍失败时，系统必须保留该次投递失败记录，便于运维追踪。
- 当同一个机器人服务多个飞书会话时，这些会话共享同一个 agent 与 workspace，因此运行状态和排队容量也是共享的。
- 当收到未通过飞书验签或不在最小 allowlist 范围内的请求时，系统必须拒绝处理，并记录安全审计事件。

## 需求 *(必填)*

### 功能需求

- **FR-001**: System MUST 接收来自飞书机器人的普通消息，并将其解析为当前 agent 上的一次运行请求。
- **FR-002**: System MUST 在首次处理某个飞书 `chat` 消息时，为该 `chat` 创建或恢复 session，并将其绑定到本地配置声明的默认 agent 与默认 workspace。
- **FR-003**: System MUST 对同一 workspace 保证任意时刻最多只有一个活动运行；后续请求必须进入 FIFO 队列。
- **FR-004**: System MUST 将运行中的关键状态至少以“已排队、已开始、执行中、已完成、已失败、已取消”的用户可理解形式回传到原飞书会话。
- **FR-005**: System MUST 在首版中以“状态变化 + 阶段性摘要 + 最终结果”的形式反馈运行进展，而不是要求逐 token 实时流式输出。
- **FR-006**: Users MUST be able to 使用 `/status` 查看当前会话对应 agent 的 workspace、当前活动运行状态、当前会话最近一次请求是否排队，以及其前方队列长度；首版不返回完整队列列表。
- **FR-007**: Users MUST be able to 使用 `/abort` 取消当前活动运行；取消不能清空后续排队请求。
- **FR-008**: System MUST 在本地配置缺失、workspace 不可用、Codex 启动失败、运行超时或进程异常退出时，返回明确失败结果并保留运行状态。
- **FR-009**: System MUST 持久化 sessions、agent_runs、run_events 和 outbound deliveries，使运行结果在网关或执行器重启后仍然可查询。
- **FR-010**: System MUST 保持“同一个机器人可服务多个飞书会话，但这些会话共享同一个 agent 与 workspace”的产品语义。
- **FR-011**: System MUST 在接收飞书 webhook 时执行验签与最小 allowlist 校验；未通过校验的请求不得进入运行链路，并应保留审计记录。

### 运维与可观测性需求 *(执行流变化时必填)*

- **OR-001**: System MUST 记录并暴露当前功能涉及的 operator-visible states，包括 session 绑定结果、run 状态迁移、队列位置、取消结果和投递失败。
- **OR-002**: System MUST 明确定义运行请求的超时、取消、失败回传和出站投递重试行为。
- **OR-003**: System MUST 记录与展示 workspace 锁、队列长度、执行器心跳和失败原因，以支持问题排查。

### 关键实体 *(涉及数据时填写)*

- **AgentConfig**: 当前本地 agent 的配置声明，包含 agent 标识、bridge 类型、默认 workspace 和运行限制。
- **Session**: 飞书 `chat` 到固定 agent/workspace 的绑定记录，包含渠道、chat 标识、绑定状态和最近活动信息。
- **RunRequest**: 由普通消息触发的运行请求，包含 session、agent、workspace、原始 prompt 和运行约束。
- **Run**: 一次排队或执行中的运行实体，包含状态、开始结束时间、失败原因和取消信息。
- **RunEvent**: 运行生命周期中的规范事件，如排队、开始、阶段摘要、完成、失败和取消。
- **OutboundDelivery**: 向飞书会话发送状态更新或结果消息的投递记录。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**: 在配置正确的前提下，用户可以在单个飞书会话中通过一次普通消息完成从发起请求到收到最终结果的完整闭环，无需访问终端或后台。
- **SC-002**: 当同一 workspace 同时收到两个请求时，系统能稳定保持单活动运行，并让后续请求进入可见排队状态。
- **SC-003**: 用户在有活动运行时发送 `/abort`，系统能在可接受时间内回传明确的取消结果，而不是无响应或状态不一致。
- **SC-004**: 用户在任意时点发送 `/status`，都能获得与持久化运行状态一致的结果说明。
- **SC-005**: 运维人员无需登录执行宿主机，即可从持久化状态判断一次运行是已完成、失败、取消还是因投递失败未成功通知用户。
