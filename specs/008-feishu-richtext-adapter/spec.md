# 功能规格说明：Feishu 稳定富文本适配

**功能分支**: `008-feishu-richtext-adapter`
**创建日期**: 2026-03-10
**状态**: 草稿
**输入**: 用户描述："为 Feishu 适配器增加稳定富文本转换层：在 agent 输出之后，由 `channel-feishu` 内部将流式输出和终态结果统一转换为 Feishu 可稳定渲染的富文本与卡片结构，避免依赖不稳定 Markdown / HTML 直出，并保持单消息卡片语义。"

## 系统影响 *(必填)*

- **受影响渠道**: Feishu
- **受影响桥接器**: Codex
- **受影响执行路径**: outbound delivery
- **运维影响**: retries, notifications, admin visibility
- **范围外内容**: 其他渠道适配器、agent 输出语义改写、结果摘要增强、队列与锁语义调整、会话记忆逻辑、调度器与外部 webhook 行为变更

## 用户场景与测试 *(必填)*

### 用户故事 1 - 流式输出稳定可读（优先级：P1）

作为在飞书中等待运行结果的用户，我希望运行中的卡片输出能够稳定显示标题、列表、代码、路径和命令等结构，而不是因为 Markdown 或 HTML 兼容性差而出现整段塌缩、标签直出或样式失真。

**优先级原因**: 这是本功能最核心的用户价值。如果运行中的内容仍然频繁渲染异常，用户对系统正在做什么的判断就会持续失真。

**独立验证方式**: 在飞书中发起一次包含标题、列表、代码块、路径和链接的普通请求，确认运行中卡片能持续更新并保持可读结构即可完整验证。

**验收场景**:

1. **Given** 一次普通请求已经进入执行并产生流式输出，**When** 输出中包含标题、列表、代码块、路径或命令，**Then** 运行中卡片应以飞书可稳定渲染的结构展示这些内容，而不是把整段输出拍平成单段普通文本。
2. **Given** 流式输出中出现飞书不稳定或不支持的 Markdown / HTML / XML 片段，**When** 系统更新运行中卡片，**Then** 系统应将这些片段转换为可读的稳定表达或安全降级文本，而不是原样透传导致渲染异常。
3. **Given** 流式输出尚未形成完整结构，例如未闭合代码块或未完成表格，**When** 系统渲染当前可见内容，**Then** 用户仍应看到稳定、不中断的可读内容，后续结构完整后允许自动收敛为更合适的展示形式。

---

### 用户故事 2 - 终态结果保持单消息结构（优先级：P2）

作为在飞书中查看最终结果的用户，我希望同一张过程卡片在完成、失败或取消后，继续以清晰分段的结构展示终态内容，而不是再收到第二条成功消息，或者看到不稳定的 Markdown 语法直接暴露出来。

**优先级原因**: 这决定了“单消息卡片语义”是否真正成立，也决定了用户回看结果时是否能快速理解结论与关键内容。

**独立验证方式**: 发起一次普通请求并等待其完成、失败或取消，确认同一张卡片切换为稳定可读的终态结果，且不会额外发送第二条成功消息即可验证。

**验收场景**:

1. **Given** 某次运行已经成功创建过程卡片，**When** 运行进入完成、失败或取消中的任一终态，**Then** 同一张卡片必须切换为终态结果卡片，并继续以稳定的结构展示终态内容。
2. **Given** 终态内容中包含标题分段、列表、代码块、链接、图片或其他富文本语义，**When** 系统切换终态卡片，**Then** 系统应优先保留这些结构化语义，并将不稳定语法归一化为飞书可稳定渲染的等价表达。
3. **Given** 过程卡片已经成功创建，**When** 运行正常结束，**Then** 用户只能看到同一条消息演进为终态卡片，不得再收到第二条成功终态消息。

---

### 用户故事 3 - 降级后仍然有可读交付（优先级：P3）

作为依赖飞书卡片查看执行结果的用户或运维人员，我希望即使卡片渲染链路遇到不兼容内容或发送失败，系统仍然给用户留下至少一条可读交付，并让运维明确知道是转换失败、更新失败还是终态增强失败。

**优先级原因**: 这不是主要体验价值，但决定了系统在异常情况下是否仍然可靠、可审计。

**独立验证方式**: 模拟过程卡片创建失败、过程卡片更新失败以及包含不支持语法的输出，确认用户最终仍能收到一条可读终态交付，且运维能区分失败阶段即可验证。

**验收场景**:

1. **Given** 输出中包含飞书无法稳定渲染的复杂语法，**When** 系统执行转换，**Then** 系统必须优先输出可读的降级结果，而不是把原始不稳定语法直接暴露给用户。
2. **Given** 过程卡片创建失败且用户侧尚无任何成功送达的消息，**When** 运行结束，**Then** 系统必须退化为单条终态交付，保证用户仍有结果可读。
3. **Given** 过程卡片已成功创建但后续更新或终态增强失败，**When** 运维检查该次运行，**Then** 运维必须能够识别失败发生的阶段、原因，以及用户最终看到的是哪一种交付形态。

### 边界与异常场景

- 当一次运行没有产生中间输出，或输出极短时，系统仍应保留过程卡片和终态卡片语义，不得因为缺少富文本结构而破坏单消息体验。
- 当流式输出包含未闭合代码块、残缺表格、截断标题或半截标签时，系统应优先保证稳定显示，再在后续内容补齐后自然收敛。
- 当输出包含飞书不支持的 HTMLBlock、未知标签或不安全标签时，系统必须转为稳定的可读文本，而不是尝试原样渲染。
- 当输出过长时，系统可以做最小必要的分段、截断或窗口控制，但不得改变用户对当前运行状态和主要内容的理解。
- 当工作区中已经有一个活动运行时，新请求继续遵守现有 FIFO 队列与单活动运行约束；本功能不得提前创建伪造的运行中卡片或修改排队语义。
- 当执行器心跳丢失、智能体进程异常退出、超时或取消发生时，系统仍按既有终态语义结束运行，但终态卡片内容必须继续通过稳定转换输出为可读结果。
- 当过程卡片更新失败时，系统不得通过新增第二条成功终态消息破坏单消息语义；仅当卡片从未成功创建时，才允许使用单条终态兜底交付。
- 当出站投递重试耗尽时，用户侧至少应保留一条已成功送达的终态交付；运维侧必须能看到失败阶段、失败原因和最终交付状态。

## 需求 *(必填)*

### 功能需求

- **FR-001**: System MUST 在 agent 输出进入 Feishu 出站链路后、消息实际发送前，对运行中输出和终态结果执行统一的富文本稳定化转换。
- **FR-002**: System MUST 对同一轮运行的流式输出和终态结果使用一致的转换语义，使用户在运行中和运行结束后看到的结构表达保持连续，不出现明显规则漂移。
- **FR-003**: System MUST 在可稳定表达的前提下优先保留标题、段落、列表、强调、引用、代码、路径、命令、链接、图片和分段信息，而不是默认退化为纯文本。
- **FR-004**: System MUST 将飞书不稳定或不支持的 Markdown、HTML、XML 或其他标记语法归一化为飞书可稳定渲染的等价表达，或降级为可读文本。
- **FR-005**: System MUST 禁止将未知、未闭合或明显不受支持的标签原样透传到用户可见卡片中，避免导致渲染异常或结果误读。
- **FR-006**: System MUST 对流式阶段的残缺结构保持容错，使未闭合代码块、未完成列表、未完成表格或未完成标题不会破坏当前卡片的整体可读性。
- **FR-007**: System MUST 在终态结果中保留单消息卡片语义；当过程卡片已成功创建时，终态增强不得通过新增第二条成功消息完成。
- **FR-008**: System MUST 仅做“稳定渲染转换”，不得主动压缩、总结、重写或替换 agent 原始语义，不得把适配层变成摘要器或内容编辑器。
- **FR-009**: System MUST 在包含复杂富文本结构的运行中，允许按飞书稳定渲染边界做分段重组，但重组后仍应保持原始内容的阅读顺序和主要语义。
- **FR-010**: System MUST 在过程卡片创建失败、过程卡片更新失败或终态增强失败时，维持现有降级策略，保证用户最终至少收到一条可读终态交付。
- **FR-011**: System MUST 保持 `ChannelAdapter` 与 `AgentBridge` 既有边界不变，使富文本稳定化属于 Feishu 渠道内部责任，而不是上移到通用执行或桥接语义层。
- **FR-012**: Users MUST be able to 在不理解 Feishu Markdown / HTML 支持边界的情况下，直接从卡片中读取稳定、清晰且不易误解的运行中与终态内容。

### 运维与可观测性需求 *(执行流变化时必填)*

- **OR-001**: System MUST 为“转换成功”“转换降级”“卡片创建失败”“卡片更新失败”“终态增强失败”“兜底终态发送”这些阶段记录明确的 operator-visible 状态和结构化日志。
- **OR-002**: System MUST 让运维能够区分“内容被稳定化转换后正常送达”“内容因不兼容而降级送达”“卡片链路失败后走兜底交付”这三类结果。
- **OR-003**: System MUST 明确说明本功能不改变锁、队列、取消、超时、心跳和会话续聊语义，任何失败只影响 Feishu 出站呈现层而不改变运行控制流。

### 关键实体 *(涉及数据时填写)*

- **Feishu Renderable Content**: 表示某次运行输出经过稳定化转换后、可安全发送给飞书卡片的用户可见内容。
- **Transformation Outcome**: 表示某次转换的结果类型，包括保留结构、局部降级、整段转义、兜底交付触发等面向用户与运维的结果语义。
- **Run Presentation**: 表示某次运行在 Feishu 侧的用户可见呈现状态，包含过程卡片、终态卡片、降级交付和是否维持单消息语义。
- **Run / Delivery / Run Event**: 沿用既有执行与投递实体，本功能仅增强其在 Feishu 渠道中的呈现解释，不改变其生命周期定义。

## 假设

- 飞书卡片正文只能稳定支持有限范围的富文本语义，不能被视为完整的 Markdown、HTML 或 XML 渲染环境。
- agent 输出可能在流式阶段暂时处于不完整状态，因此稳定显示优先于严格保留瞬时语法形态。
- 用户更重视结果是否稳定可读，而不是逐字符保留原始语法写法。
- 当卡片创建已成功时，维持单消息卡片语义的优先级高于补发第二条更漂亮的成功消息。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**: 在包含标题、列表、代码块、链接、路径或命令的代表性运行中，95% 的运行中卡片更新能保持用户可读结构，而不会出现大段标签直出、整体塌缩或明显结构丢失。
- **SC-002**: 在过程卡片创建成功的运行中，100% 的终态结果继续落在同一条用户可见消息上，不额外产生第二条成功终态消息。
- **SC-003**: 在包含飞书不稳定或不支持标记语法的代表性运行中，95% 的终态交付仍被评估为“主要内容可读且不易误解”。
- **SC-004**: 当富文本转换或卡片增强链路发生失败时，运维在不登录宿主机的前提下可以从持久化状态或日志中判断失败阶段、失败原因和最终交付路径。
- **SC-005**: 引入本功能后，现有工作区串行化、取消、超时、心跳失效和会话续聊相关用户可见语义不发生回归。
