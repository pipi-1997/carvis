# 合同：Feishu 富文本稳定化转换

## 1. 输入边界

- 输入来自 `channel-feishu` 出站 sender，而不是 `gateway` 或 `bridge-codex` 直接传入的 Feishu 专属结构。
- 输入文本必须是当前用户可见的累计完整文本。
- 转换模式分为：
  - `streaming`
  - `terminal`

## 2. 主流程契约

- 转换必须遵循统一的“解析 -> 归一化 -> 映射”主流程。
- `streaming` 与 `terminal` 只允许在策略上有少量差异，不允许形成两套独立且不兼容的渲染规则。
- 转换结果必须保持原始阅读顺序，不得重排主要语义。

## 3. 语义保留契约

- 对以下结构，系统应优先保留为飞书可稳定渲染的等价表达：
  - 标题
  - 段落
  - 有序 / 无序列表
  - 强调
  - 引用
  - 代码块
  - 行内代码
  - 路径与命令
  - 链接
  - 图片
  - 分段
- 对标题类结构，不得单纯依赖原始 Markdown 标题语法稳定生效；应转换为稳定的 section 表达。

## 4. 降级契约

- 对飞书不稳定或不支持的 Markdown / HTML / XML 结构，系统必须执行最小必要降级。
- 未知标签、未闭合标签、明显不支持的标签、HTMLBlock 和危险标签不得原样透传到最终用户可见卡片。
- 流式阶段若结构尚未闭合，系统必须优先输出稳定可读结果，而不是等待完整结构后才显示。

## 5. 非目标

- 转换层不得做摘要、润色、重写、去套话或结果编辑。
- 转换层不得改变终态消息数量规则。
- 转换层不得引入新的运行状态、队列状态或心跳状态。
